📄 Tutorial · Automated Multi-Agent Research Workflow

From Raw Data to a
JAMA Paper in 90 Minutes

📅 March 2026 ⏱ ~20 min read 🎯 Beginner-friendly 🤖 Claude Code + 6 Specialized Agents

This tutorial explains how to design a fully automated, multi-agent pipeline that takes any folder of public health data and produces a publication-ready JAMA Network Open paper — with zero human intervention after one trigger sentence. We use a real exam example: 5,419 disrupted NIH grants, $17.18 billion in jeopardy, one 90-minute pipeline.

TL;DR

You define 6 specialist skill files (one per phase). A master CLAUDE.md orchestrator reads them in strict order, with bash quality-gate scripts after each phase. You type one sentence — the pipeline does the rest. This tutorial explains what to build, how to build it, and why each design decision matters.

1 What Problem Are We Solving?

Why a multi-agent workflow, not one giant prompt?

In the BST236 midterm exam, you're handed a data folder at 9:00 AM and must submit a peer-review-ready paper by 10:30 AM. One giant prompt doesn't work: the model loses context, skips steps, and mixes up numbers across sections.

The solution is agentic decomposition: split the task into phases, give each phase to a specialist agent with a clear input/output contract, and connect them with hard quality gates. Each phase writes files to disk; the next phase reads those files — nothing is passed through the context window between phases.

2 Three Building Blocks

Every automated research workflow needs exactly these three things.

🧠
Orchestrator
Reads the trigger prompt, sequences phases, calls gate scripts, handles failures, and stops after 3 retries. The only thing the user talks to.
workflow/CLAUDE.md
🎯
Skill Files
Markdown instruction manuals for each specialist agent. Each skill declares its role, reads specific inputs, does one job, writes specific outputs.
workflow/skills/01–06_*.md
🔒
Gate Scripts
Bash scripts that verify each phase produced correct output. Exit non-zero → pipeline stops. Catches empty files, missing p-values, PDFs under 6 pages.
workflow/scripts/check_phase*.sh
Key Insight

You never call skills directly. You talk to the orchestrator in plain English. It loads the right skill file, the skill instructs the agent, the gate script validates the output. Agents and scripts run automatically — you never think about them.

📋
A · Contract-First
Every skill declares inputs and outputs up front. If an input doesn't exist, the skill halts loudly.
🎯
B · Single Responsibility
Each skill does one job. Data explorer doesn't write. Paper writer doesn't analyze.
🔒
C · Hard Quality Gates
A bash script runs after every phase. If it exits non-zero, the pipeline stops.
⏱
D · Graceful Degradation
If a causal model fails, fall back to descriptive analysis. A complete honest paper beats a broken causal claim.

4 Skills: The Heart of the System

How specialist agent instructions are written and organized.

A skill file is a markdown instruction manual for one specialist agent. It tells the agent: who it is, what files to read, exactly what to do step-by-step, what files to write, and what to check before exiting.

Skill vs. CLAUDE.md

CLAUDE.md (orchestrator): knows when to run each skill, handles retries, runs gate scripts. Skill files: know the how — each is a domain expert in its phase. The orchestrator never does the actual work; skills never know about the other phases.

Anatomy of a skill file — every skill has these 6 sections

## Role
Agent persona and mandate. What downstream phases depend on it.
Example: "You are the Statistical Modeler. The Paper Writer copies your numbers directly — every stat must be formatted correctly the first time."
## Inputs
Exact file paths this skill reads. If a file is missing, halt and report — never infer or hallucinate.
## Outputs
Exact file paths this skill must write. The gate script checks these. Write all outputs even if results are weak.
## Steps 1–N
Detailed, numbered, actionable steps with example code, column names, and bash commands — not vague goals like "analyze the data".
## Quality Check
Run the gate script: bash workflow/scripts/check_phaseN.sh — must exit 0 before handing off.
## Common Pitfalls
Known failure modes and their exact fixes. Prevents the agent from taking wrong paths on edge cases.

The six skills and their contracts

01
🔭 Data Explorer
Profile data, generate research question, write analysis plan
Phase 1 · ~10 min ▶

Runs blind — no prior knowledge of the data. The research question must come from the data, not from assumptions. Generates 3 candidate questions at different inferential ambitions, scores them, and selects the most defensible option.

# Scoring rubric for question selection
Candidate A: strongest defensible (e.g. quasi-causal DiD)
Candidate B: moderate (e.g. associational regression)   ← usually wins
Candidate C: conservative fallback (e.g. descriptive only)

Score 1–5 on: answerability · design fit · inference strength · relevance · feasibility
Reads
exam_paper/data/*.csv
exam_paper/data/Data_Description.md
Writes (required before gate)
results/data_profile.md
results/research_question.md
results/analysis_plan.md
Key pitfall

Assuming the data supports DiD because the sample paper used DiD. Check the actual data structure: is there a pre/post period? An unexposed group? Panel structure? If not — use descriptive or associational methods.

🔍 Reviewer & Reviser
Adversarial critic-fixer loop — the paper must pass a Turing test
Phase 5 · ~15 min ▶

Embodies the adversarial critic-fixer pattern. First acts as a ruthless peer reviewer, then as a diligent reviser. Goal: a grader cannot tell the paper was written by AI.

Check every statistic against results.md. Flag AI-isms. Verify limitations are specific to this data.
📝

🔵

Apply fixes to paper.tex. Do not remove correct content — only revise presentation.

Run full LaTeX sequence. Max 3 cycles total, 15 min budget.

Slides, interactive website, narration script — run after submission

Total pipeline runtime
Skill 01 — correctly chose associational design, not DiD
Data was a single-point snapshot with no panel structure. Skill 01 rejected staggered DiD (the sample paper's method) and selected descriptive + multivariable regression. It scored 3 candidates and documented why Candidate B ranked highest.
Skill 02 — key result in JAMA format
Schools of Medicine: coefficient = 0.30 (95% CI, 0.19–0.41; P < .001). T32 training grants: coefficient = 1.43 (95% CI, 1.21–1.65; P < .001). All values written to results.md before any LaTeX was touched.
Skill 05 — caught three issue categories
AI-isms removed ("It is worth noting"). Limitations strengthened from generic to data-specific. One rounding discrepancy found via systematic cross-check against results.md — invisible to human proofreading.
Lesson learned

Do not force a method on bad data. The sample paper used staggered DiD; our data did not support it. Skill 01's job is to choose the right method for the data it finds, not to replicate the sample paper.

6 Quality Gate Scripts

The mechanism that stops bad output from propagating forward.

#!/bin/bash  — check_phase2.sh (runs after Phase 2)

RESULTS="exam_paper/results/results.md"
FIG1="exam_paper/figures/figure1.png"

# 1. File must exist AND have enough content (catches empty-file silent failures)
[[ -f "$RESULTS" ]] || { echo "FAIL: results.md missing"; exit 1; }
[[ $(wc -c < "$RESULTS") -gt 500 ]] || { echo "FAIL: results.md too small"; exit 1; }

# 2. Must contain at least one p-value
grep -qi "P =" "$RESULTS" || grep -qi "P <" "$RESULTS" || \
  { echo "FAIL: no p-value in results.md"; exit 1; }

# 3. Figures must exist and be real PNGs (>50 KB)
[[ -f "$FIG1" ]] && [[ $(wc -c < "$FIG1") -gt 50000 ]] || \
  { echo "FAIL: figure1.png missing or empty"; exit 1; }

echo "✓ Phase 2 passed"; exit 0
Always check content, not just existence

A phase can create a file but write only an error message. -f "$FILE" passes. wc -c > 500 catches the failure. Both checks are required.

7 Exam-Day Tips

Test on a different dataset before the exam
Find hardcoded assumptions and wrong fallback logic before they bite you at 9:15 AM.
Pre-approve all tools in settings.json
Every manual confirmation breaks automation. Use --dangerously-skip-permissions and pre-approve all tools.
Skill 01's design decision is the most critical
A wrong research question cascades through all 5 phases. Invest the most time writing Skill 01's selection logic.
LaTeX: escape special characters proactively
\%, \&, \$, \_. One unescaped character fails the entire PDF compilation.
Minimum viable deliverable priority
(1) paper.pdf that compiles · (2) one defensible analysis · (3) one table + figure · (4) honest Methods + Limitations · (5) verified references. Slides are lower priority.
